## 武库检索模块抽取与派生素蕴修复交接

> 将「读取用户已获得山河器详情」从 `WukuScanner` 中抽取为独立 `WukuReader` 模块，并修复 `derived_affixes` 全为空的问题，为后续并行的「孔位培养读取器」奠定统一架构。

### 1. 工作路径

#### 工具使用统计
| 工具名称 | 调用次数 | 主要用途 |
|---------|---------|---------|
| Read | 约 15 次 | 读取 `wuku_scanner.py`、现有交接文档格式、扫描结果 JSON、单元测试 |
| Write | 约 4 次 | 新建 `shq/scanner/readers/` 包、`wuku_reader.py`、单元测试、本交接文档 |
| Edit | 约 15 次 | 重写 `_select_quality`、扩展 grid ROI、透传 `derived_tags`、更新 CLI 入口 |
| Bash | 约 20 次 | 运行单品质扫描、完整扫描、单元测试、截图 OCR 分析 |

#### 执行阶段
- **阶段一：问题定位**
  - 目标：确认 `derived_affixes` 为何全为空。
  - 关键动作：OCR 分析详情面板发现「专属词条」区域只展示触发条件/数值，不展示派生素蕴名称；进一步截图左侧网格 item 文字，确认派生素蕴名字显示在等级下方。

- **阶段二：修复派生素蕴识别**
  - 目标：让 `derived_affixes` 正确填充。
  - 关键动作：
    - 将 `_detect_cells()` 的网格 ROI 从 `y=140~700` 延伸至 `y=140~750`，避免最底行 item 的派生素蕴小字被截断；
    - 将网格识别到的 `derived_tags` 加入 `PendingDetail` 并在 `_consume_detail()` 透传回 `CellInfo`；
    - `_parse_detail_impl()` 初始化 `derived` 时直接使用 `cell.derived_tags`。

- **阶段三：修复品质下拉框误切「全部」**
  - 目标：解决扫描指定品质时误切到「全部」的问题。
  - 关键动作：
    - 明确下拉框基础顺序为「全部/朴素/精巧/瑰丽/绝世」；
    - 用 OCR 识别当前选中品质，根据「展开后排除当前项、其余四项保持基础顺序」的规则计算目标品质点击位置；
    - 用固定坐标点击下拉框按钮与计算出的选项位置，并加入验证重试。

- **阶段四：抽取 WukuReader 模块**
  - 目标：将「导航 + 读取用户已获得山河器详情」独立为可复用模块。
  - 关键动作：
    - 新建 `shq/scanner/readers/` 包；
    - 实现 `WukuReader.read()` 与 `WukuReader.read_quality()`；
    - `shq/cli.py` 与 `scan_quality.py` 改为调用 `WukuReader`；
    - `WukuScanner` 退化为底层网格扫描/解析实现。

- **阶段五：回归验证**
  - 目标：确保重构后扫描结果与之前一致，派生素蕴正确。
  - 关键动作：
    - 单元测试：`python -m pytest tests/ -q` 50 个全部通过；
    - 真实扫描：`python main.py --scan-all-owned --ocr-backend rapidocr`，总计 16 个山河器，派生素蕴识别到 5 个。

### 2. 关键发现

#### 任务范围内的发现
- `derived_affixes` 为空不是因为白名单错误，而是因为：
  1. 网格 ROI 底部截断，导致最底行 item 的派生素蕴小字未被 OCR；
  2. 即使 OCR 识别到，`derived_tags` 在传给详情解析的 `PendingDetail` 时丢失。
- 右侧详情面板的「专属词条」只显示触发条件（如「五行属性为金时激活属性」）和数值，**不显示派生素蕴名称**；名称只在左侧网格 item 等级下方显示。
- 品质下拉框的选项顺序是固定的「全部/朴素/精巧/瑰丽/绝世」，但展开时会排除当前选中项，导致目标品质的像素位置随当前选中项变化，不能硬编码。
- 旧代码中 `QUALITY_DROPDOWN_BUTTON` 与「全部」选项坐标重合，是当前选中为「全部」时误触的隐患。

#### 延伸发现
- `WukuScanner` 原本同时承担窗口/导航/UI 交互、截图、OCR 解析、业务规则等多重职责，不利于后续扩展「孔位培养读取器」等并行模块。
- 截图文件名在多品质扫描时存在冲突（各品质都从 `step=0` 开始），已通过在文件名中加入品质前缀解决。

### 4. 核心结论

本次任务完成了两件事：

1. **派生素蕴识别已修复**：`derived_affixes` 现在能正确从左侧网格 item 文字中读取。以当前用户数据为例，16 个已获得山河器中有 5 个带有派生素蕴（均为「起势」），与游戏界面一致。
2. **武库检索已抽取为独立模块**：`shq/scanner/readers/wuku_reader.py` 中的 `WukuReader` 负责「读取用户已获得山河器详情」的完整流程，与底层 `WukuScanner` 解耦。后续并行的「孔位培养读取器」可遵循同一目录和接口模式实现。

### 6. 假设前提

- 游戏窗口客户区固定为 1334×750；品质下拉框坐标和 grid ROI 基于此分辨率标定。
- 用户已安装 `rapidocr-onnxruntime` 并使用 RapidOCR 后端。
- 派生素蕴名称（如「起势」「承势」「金实」等）出现在左侧网格 item 的等级下方，若游戏 UI 改版则需重新标定识别区域。
- 品质下拉框选项的基础顺序保持「全部/朴素/精巧/瑰丽/绝世」，且展开时排除当前选中项。
- 用户理解自动化点击存在封号风险，自愿承担。

### 7. 下一阶段建议

- 如后续需要保存中间截图审计，可统一在 `WukuReader` 层管理输出目录生命周期。
- 考虑把 `NavigationController` 与 `WukuNavigator` 统一，减少两套导航器的维护成本。
